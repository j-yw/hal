## Codebase Patterns
- Cloud config lives in internal/cloud/config; use config.Parse(data) or config.Load(path) for YAML cloud.yaml handling
- Use template.CloudConfigFile constant for the cloud.yaml filename
- ValidationErrors type implements error interface and collects multiple field-level failures with path/rule/remediation
- Secret detection walks raw YAML map recursively; checks both key names AND string values for secret-bearing URL query parameters
- Profile.Wait uses *bool (pointer) to distinguish omitted from explicit false — follows the pointer-yaml-defaults pattern from AGENTS.md
- ArtifactGroup (state/reports/all) follows the same typed string enum pattern as WorkflowKind and RunStatus — use NewArtifactMetadata(kind) to build per-workflow metadata
- BundleAllowlist is for input uploads; artifact groups are for output pull — they overlap but are distinct concerns (auto-state.json is output-only)
- WorkflowArtifactGroups maps workflow kinds to their default artifact groups: run→[state], auto/review→[state,reports]
- Use RegisterCloudFlags(cmd) from cmd/cloud_flags.go to add the shared cloud flag set to any workflow command; ValidateCloudFlags(f) checks --detach/--wait conflict
- Doctor command uses httpDoer interface for testable HTTP calls — inject mockHTTPClient in tests to avoid real network requests
- Use config.Resolve(ResolveInput{...}) to compose cloud runtime config from CLI/env/.env/YAML/inferred/hard defaults with deterministic precedence
- ResolvedConfig is the same shape for all workflow kinds (run/auto/review) — only WorkflowKind field differs
- When CloudEnabled=false, resolver returns hard defaults without loading cloud tiers — verifiable via golden tests
- Load() wraps os.ReadFile error with fmt.Errorf %w — use errors.Is(err, os.ErrNotExist) not os.IsNotExist for wrapped errors
- buildHalCommand(kind, mode) dispatches worker commands by WorkflowKind: run→"hal run", auto→"hal auto --mode <mode>", review→"hal review" — mode only applies to auto
- ListRuns(ctx, limit) is the Store method for listing runs — ordered by updated_at DESC with LIMIT; use cloudListStoreFactory for test injection
- Cloud list command follows the same pattern as status/cancel: separate file (cloud_list.go), testable runCloudList function, factory variable, init() wiring
- For `hal run --cloud`, use executeRunCloud() as an early return guard in runRunWithWriter; returns (handled bool, err) so the local flow continues when --cloud is unset
- runHalRunCloud() is the testable function for hal run --cloud — accepts CloudFlags, halDir, baseDir, storeFactory, configFactory, and io.Writer
- Wait/detach behavior: --detach overrides to no-wait, --wait overrides to wait, otherwise use resolved.Wait from config.Resolve()
- pollRunUntilTerminal() polls store.GetRun until IsTerminal(); uses runCloudPollInterval variable for test override
- JSON response uses camelCase field names (runId, workflowKind, status) matching the PRD contract
- writeCloudError(out, jsonOutput, msg, code) in JSON mode writes JSON to output and returns nil — test JSON error cases by checking output content, not Go error return
- Cloud logs JSON uses cloudLogsResponse{run_id, status, events[]} — follow mode collects all events and emits single JSON document at terminal/cancel
- Cloud cancel pre-checks run.Status.IsTerminal() before RequestCancel — terminal runs get `already_terminal` status with `terminal_status` field; non-terminal runs proceed with cancel intent
- Cloud pull uses `MatchesArtifactPatterns` (not `IsBundlePathAllowed`) for artifact-group-based filtering — reports are valid pull artifacts despite being excluded from uploads
- `decompressBundleFiles` expects gzip-compressed `path\x00size\x00content` frame format matching `compressBundleFiles`
- Pull local-change detection uses `bytes.Equal` on file content — identical files pass silently, only truly changed files trigger the `--force` requirement
- `mapAuthProfileStatus()` maps internal AuthProfileStatus to user-facing states: pending_link→missing, linked/invalid/revoked pass through. Use this for any user-facing profile status display
- Auth commands (status, validate) use camelCase JSON fields (profileId, validatedAt, lastValidatedAt) — normalize all auth command JSON to camelCase when updating
