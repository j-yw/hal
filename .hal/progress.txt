## Codebase Patterns
- `RunLoop` uses `select` on poll/reconcile/timeout tickers + `ctx.Done()` — V1 behavior: long `ProcessOne` blocks all ticks. Reconciler/Timeout are optional (nil-guarded), unlike required pipeline dependencies.
- For cmd-level `runCloudWorker` tests, use `injectWorkerTestFactories(t)` to override package-level store/runner factories with mocks — the mock store must implement `ClaimRun` returning `ErrNotFound`.
- Cloud worker factory vars use `cloudWorkerStoreFactory` (store) and `cloudWorkerRunnerFactory` (runner) — override in tests, restore in cleanup. Runner factory takes `deploy.Config` input, store factory is `func() (cloud.Store, error)`.
- Use `resolveWorkerSandboxImage(flagImage)` to ensure `ProvisionConfig.Image` is always non-empty — returns `defaultWorkerSandboxImage` (`"ubuntu:22.04"`) when flag is empty.
- Postgres/Turso ClaimRun uses `FOR UPDATE SKIP LOCKED` subquery pattern for atomic claim — modifications go in the outer UPDATE SET clause.
- Integration tests behind `//go:build integration` tags need `go vet -tags=integration` to verify compilation since default `go test ./...` skips them.
- Worker lifecycle JSON contract assertions should come from shared fixtures (`workerLifecycleJSONContractFixtures` for status/logs/pull) and use `mustDecodeWorkerLifecycleJSONOutput` so required keys stay camelCase and matrix-wide.
- For worker lifecycle JSON assertions, use `assertWorkerLifecycleCheckpointJSONContract` for status/logs/pull and `assertWorkerLifecycleCanonicalJSONContract` for migrated workflow submit payloads (`run/auto/review`) so snake_case alias regressions fail fast.
- New cmd integration suites should start with a dedicated `*_integration_test.go` scaffold file that defines a top-level `Test<Feature>Integration` entrypoint; expand scenarios in additional tagged files.
- For workflow-matrix integration suites, define one shared flow-step template (e.g., setup/submit/status/logs/pull/cancel) using placeholders like `<workflow-command>` and `<run-id>`, then resolve placeholders per scenario to avoid drift.
- For cross-adapter lifecycle integration, define `workerLifecycleAdapterFixtures` once (`postgres`, `turso`) and run scenarios through a shared `runWorkerLifecycleAdapterMatrix` helper with `adapter/scenario` subtest names and per-case harness teardown via `t.Cleanup`.
- Worker lifecycle harness stores should override `TransitionRun` + `TransitionAttempt` and expose query helpers (`RunTransitions`, `AttemptTerminalizations`, `AttemptTerminalizationCount`, `SnapshotRefs`) so scenarios can assert lifecycle invariants deterministically without poking raw maps.
- For deterministic success-path worker lifecycle scenarios without a live worker loop, seed harness state in order: `TransitionRun` queued→claimed→running→succeeded, `CreateAttempt` + terminal `TransitionAttempt(succeeded)`, then persist a latest snapshot (`PutSnapshot`) before `UpdateRunSnapshotRefs`; this keeps `cloud pull` assertions deterministic and restorable from harness state.
- For deterministic cancel-path worker lifecycle scenarios, seed run state to `running` with one active attempt before `cloud cancel`; then record exactly one terminal `TransitionAttempt(..., canceled, ...)` and assert both persisted `cancel_requested/status=canceled` and `AttemptTerminalizationCount(runID) == 1`.
- For deterministic lease_lost worker lifecycle scenarios, seed a `running` run with one active attempt, emit a `lease_lost` timeline event, transition the attempt exactly once to `failed` with `error_code=lease_lost`, then transition the run `running->failed`; assert `AttemptTerminalizationCount(runID) == 1` and no active attempt remains.
- For deterministic profile_revoked worker lifecycle scenarios, drive terminalization through `HeartbeatService.Renew` with a store wrapper that forces `ReleaseAuthLock` to return `ErrNotFound`; assert renew returns `ErrProfileRevoked`, terminal attempt reason is `profile_revoked`, and `AttemptTerminalizationCount(runID) == 1`.
- Use `h.SeedTimelineEvents(t, runID, ...)` for lifecycle log fixtures instead of direct `h.Store.InsertEvent` calls — it centralizes default event IDs/run wiring and keeps payload pointers cloned for mutation-safe assertions.
- Capture/restore package-level lifecycle factories with `snapshotCloudLifecycleHarnessFactories` + `restoreCloudLifecycleHarnessFactories`; harness setup must register teardown via `t.Cleanup` to prevent global factory leakage across integration subtests.
- `TransitionRun(claimed -> queued)` is valid for re-queuing runs in tests to simulate retry scenarios.
- `scanRun` reads all RETURNING columns including `attempt_count`, so adding SET clauses to the UPDATE is automatically reflected in the returned Run struct.
- Turso store tests are regular unit tests (no build tags) using `modernc.org/sqlite`, while Postgres store tests require `//go:build integration` and `HAL_TEST_POSTGRES_DSN`.
- After US-001/US-002 made store adapters atomically increment `attempt_count`, ClaimService must use `run.AttemptCount` directly (not `+1`) for `Attempt.AttemptNumber` since the store already incremented it.
- Mock `claimedRun()` in claim_test.go represents the run as returned by `ClaimRun` (post-increment), so `AttemptCount` should reflect the incremented value (e.g., 1 for first claim, not 0).
- Use `cloud.ShellQuote(s)` from `internal/cloud/shell.go` for safe single-quote shell escaping — wraps in `'...'` with `'` → `'\''` replacement. The old `escapeShellSingleQuote` in auth_materialization.go only escapes inner content without wrapping.
- `CollectSandboxBundle` uses `MatchesArtifactPatterns` (workflow-aware filtering) not `IsBundlePathAllowed` (input allowlist) — these are two different filtering policies for worker-side vs client-side collection.
- When building shell commands with `ShellQuote`, wrap the entire path argument (e.g., `ShellQuote("/workspace/" + relPath)`) not just part of it.
- Use `mockCollectRunner` + `sandboxFS` pattern for testing sandbox exec operations without real infrastructure.
- `readSandboxFile` uses `stripBase64Whitespace` to strip all whitespace from base64 output before decoding — handles wrapped output from non-GNU base64 implementations and trailing newlines.
- `base64Cmd` is a package-level variable (`"base64 -w0"`) used by `readSandboxFile` — mock runners must use `base64Cmd` (not hardcoded `"base64 -w0"`) when matching command prefixes.
- Bundle framing uses `path\x00size\x00content` (null-terminated path, null-terminated decimal size, raw content bytes) plus gzip — `CompressBundle` (worker-side) and `compressBundleFiles` (client-side) produce identical framing consumed by `decompressBundleFiles`.
- Use `ComputeSandboxBundleHash(records)` for worker-side snapshot SHA — it converts `SandboxBundleRecord` → `BundleManifestRecord` via `NewBundleManifestRecord` then delegates to `ComputeBundleHash`. Never hash compressed payload bytes for the snapshot SHA field.
- `GitHubPRCreator(r, sandboxID, authDir)` returns a `PRCreator` closure — use `mockPRRunner` in tests to capture and assert the executed command text without real sandbox infrastructure.
- `parseGHPROutput` extracts the PR URL from `gh pr create` stdout by scanning the last non-empty line — `gh` may emit progress text before the URL.
- `WorkerPipeline` uses config-struct constructor pattern (`WorkerPipelineConfig` → `NewWorkerPipeline`) with explicit nil/empty checks for required dependencies (Store, Runner, WorkerID, Claim).
- `ProcessOne` maps `ErrNotFound` from `ClaimAndLock` to `ErrNoWork` — callers should use `IsNoWork(err)` to detect empty queue, not `IsNotFound`.
- Worker tests use `workerMockStore` embedding `mockStore` plus `workerMockRunner` implementing `runner.Runner` — reuse this pattern for future worker-related tests.
- `executeAttempt` setup order is deterministic: TransitionRun(claimed→running) → Provision → Bootstrap → AuthMaterialization → Preflight. Tests use a shared `callLog` to verify ordering.
- `AuthMaterializationService.Materialize` is a no-op when `AuthProfile.SecretRef` is nil — call-order tests must provide a profile with a non-nil `SecretRef` to observe auth materialization exec calls.
- Use `newTestWorkerPipeline(t, store, rnr)` helper to construct a fully-wired pipeline with all services for worker tests — avoids duplicating service creation boilerplate.
- `workerMockRunner` supports `execOverrides` map (command prefix → result/error) for per-command exec failure injection without affecting other commands.
- `handleSetupFailure(ctx, runID, attemptID, fromRunStatus, stage, cause)` transitions both attempt and run to failed — uses `fromRunStatus` to handle failures before and after the run reaches `running` status.
- `transitionAttemptCall` struct is defined in `claim_test.go` and shared across all test files in `package cloud` — don't redeclare it in new test files.
- `WorkerPipeline` computes `workingBranch := WorkingBranch(run.ID)` per attempt and propagates it with `AttemptNumber`, `GitUsername`, `GitPassword` to `BootstrapRequest` — use `newTestWorkerPipelineWithOpts` with `workerMockGitOps` to test GitOps propagation.
- `WorkerPipelineConfig` requires `Checkpoint *CheckpointService` — all tests constructing `WorkerPipelineConfig` must include it.
- `WorkerPipelineConfig` requires `Heartbeat *HeartbeatService` — all tests constructing `WorkerPipelineConfig` must include it. `HeartbeatInterval` defaults to 10s; tests use 20-50ms for fast ticks.
- `startHeartbeat` returns `<-chan struct{}` (done channel) — defer `stopHeartbeat(); <-done` to guarantee goroutine cleanup. Heartbeat starts after `TransitionRun(claimed→running)`.
- Use `slowProvisionRunner` (gate-channel pattern) to simulate long setup for concurrent behavior testing — blocks `CreateSandbox` until gate is closed.
- `workerMockStore` fields accessed from goroutines (`heartbeatCalls`) must use `sync.Mutex` — heartbeat goroutine writes while test goroutine reads.
- `startHeartbeat` calls `cancel.CheckAndCancel` before `heartbeat.Renew` on each tick — when cancel is detected, the tick skips Renew and the heartbeat goroutine exits. This means `GetRun` is called twice per tick (cancel check + heartbeat auth check).
- `WorkerPipelineConfig` requires `Cancel *CancellationService` — all tests constructing `WorkerPipelineConfig` must include it.
- `callLog` has thread-safe `record()` (with mutex) and `snapshot()` method — use `snapshot()` when heartbeat goroutine may still be running, use `log.calls` directly only after `ProcessOne` returns.
- `startHeartbeat` returns `(*heartbeatResult, <-chan struct{})` — the `heartbeatResult.LeaseLost` flag signals when `Renew` returned `ErrLeaseExpired`. Stop heartbeat and drain done channel before reading the result.
- `handleLeaseLost` transitions run to failed and performs cleanup (auth lock release, sandbox destroy) but does NOT emit `TransitionAttempt` — the `HeartbeatService.emitLeaseLostAndTerminate` already did that.
- `workerMockStore.heartbeatErr` + `heartbeatErrAfter` controls when `HeartbeatAttempt` returns an error — set `heartbeatErrAfter=0` for immediate lease loss, or `>0` to simulate loss after N successful renewals.
- `workerMockRunner.destroySandboxCalls` (mutex-protected) tracks sandbox cleanup assertions. `workerMockStore.releaseAuthLockCalls` (mutex-protected) tracks auth lock release.
- `workerMockStore.profileRevoked` controls auth profile status via `GetAuthProfile` — set with mutex. When true, `HeartbeatService.Renew` returns `ErrProfileRevoked`.
- `handleProfileRevoked` does NOT release auth lock (heartbeat already did) — differs from `handleLeaseLost` which DOES release it.
- Cleanup handlers (`handleLeaseLost`, `handleProfileRevoked`, `handleSetupFailure`, `handleExecutionResult`) use `context.WithTimeout(context.Background(), cleanupTimeout)` — never the parent context, which may be canceled during shutdown.
- Use `ctxTrackingStore`/`ctxTrackingRunner` wrappers to verify context liveness at call time in tests.
- `WorkerPipelineConfig` requires `Execution *ExecutionService` — all tests constructing `WorkerPipelineConfig` must include it.
- `handleExecutionResult` maps exit code 0 → attempt succeeded + run succeeded, non-zero → attempt failed (non_retryable) + run failed. Exactly one attempt and one run transition per outcome.
- When using `execOverrides` to control execution exit codes in tests, use `"hal run"` (or the specific workflow command) as the prefix — `"hal "` is too broad and also matches `"hal init"` from bootstrap.
- Use `releaseAuthLockBestEffort(ctx, authProfileID, runID)` to release auth locks in cleanup paths — tolerates `ErrNotFound` (already released) and wraps other errors. Same pattern as `CancellationService.CheckAndCancel`.
- `workerMockStore.releaseAuthLockErr` controls auth lock release error behavior in tests — set to `ErrNotFound` to simulate already-released locks or to a custom error for failure scenarios.
- `WorkerPipelineConfig` requires `Snapshot *SnapshotService` — all tests constructing `WorkerPipelineConfig` must include it. Use `NewSnapshotService(store, SnapshotServiceConfig{IDFunc: ...})` with IDFunc for snapshot ID generation.
- `finalizeSnapshot` collects via `CollectSandboxBundle`, compresses via `CompressBundle`, hashes via `ComputeSandboxBundleHash(records)`, persists via `SnapshotService.StoreSnapshot` — all best-effort (error ignored).
- Mock runner `execOverrides` for sandbox collection need `"find /workspace/.hal"` (listing) and `base64Cmd` (content capture) prefixes — use full command strings for per-file targeting in multi-file tests.
- `WorkerPipelineConfig` optional PR fields: `PRCreate *PRCreateService`, `PREnabled bool`, `PRCreatorFunc func(sandboxID, authDir string) PRCreator` — all nil/false by default, no validation required.
- `maybeCreatePR` gates on 5 conditions: `PREnabled`, `PRCreate != nil`, `PRCreatorFunc != nil`, `workflowKind ∈ {auto, review}`, `prContext != nil`. `WorkflowKindRun` never creates PRs.
- `AuthMaterializationService.AuthDir()` exposes the configured auth directory for callers like `executeAttempt` that build `prContext`.
- Use `testClaimedRunWithKind(kind)` helper when testing workflow-kind-dependent behavior to avoid duplicating run construction.
- Use `trackingPRCreatorFactory` pattern for verifying whether PR creation was invoked and capturing the `PRCreateRequest`.
- Use `newTestWorkerPipelineWithPR(t, store, rnr, &prOpts{...})` to test PR-enabled pipelines without modifying the default test helper.
