# Ralph Progress Log

## Codebase Patterns

- Verify-v1 shared result types and constants live in `internal/verify`; use `SchemaVersion = "verify-v1"` and keep JSON field/tag plus round-trip tests beside the types.
- Verification config lives in `internal/verify`; `LoadConfig(dir)` reads `.hal/config.yaml` `verify.checks`, defaults omitted `required` to true, and normalizes omitted/empty `workDir` to the absolute project root.
- Machine contract docs belong in `docs/contracts/` and should be registered in `cmd/contracts_doc_test.go`; add field/value assertions there when a doc must stay synchronized with code constants.
- Contract example payloads live in `docs/contracts/examples/`; reference them from the contract doc and decode them in `cmd/contracts_doc_test.go` with `decodeStrictJSONExample` against the exported schema type.
- For verify-v1 schema tests, assert exact marshaled JSON keys and include zero-valued fixtures so accidental `omitempty` or required-field omissions fail tests.
- Verify shell execution lives in `internal/verify.Run`; it uses each configured `ShellCheck` command/workDir/timeout and aggregates `CheckResult` values into verify-v1 summary/status fields.
- Optional verify checks keep their concrete per-check status (`fail`, `timeout`, or `missing`) and add `Warning` entries so the top-level status becomes `warn` instead of `fail`.
- Verify shell timeout execution uses platform-specific `internal/verify/sysproc_*.go` helpers; set both `cmd.SysProcAttr` and `setupProcessCleanup(cmd)` so Unix timeouts kill shell child process groups.
- Verify execution tests use `helperCommand`/`TestVerifyHelperProcess` in `internal/verify/run_test.go` to simulate shell behavior deterministically; extend helper subcommands instead of relying on platform-specific shell utilities.
- Verify artifact capture writes non-empty stdout/stderr to `.hal/reports/verify/<check-id>-stdout.txt` and `<check-id>-stderr.txt`; `LoadConfig` sets `Config.ProjectRoot` so artifacts stay rooted at the project directory.
- Factory run verification metadata uses `factory.VerificationRecord` with `internal/verify` summary and artifact reference types; keep `factory-status-v1` docs, examples, and doc-code sync tests updated with the run record shape.
- `hal verify --json` emits the verify-v1 result to stdout before returning a typed non-zero exit for `status=fail`; keep the wrapped error nil so machine consumers get parseable JSON without extra error text.
- CLI command documentation should be authored in Cobra `Long`/`Example`, locked with command help tests, and regenerated with `make docs-cli`; use `make docs-check` to verify `docs/cli/` has no drift.
