## Codebase Patterns
- Deployment infrastructure lives in `internal/cloud/deploy/` — Config, store factory, health handler, and smoke checks
- `deploy.LoadConfig(getenv)` accepts a `func(string) string` for testability — pass `os.Getenv` in production, inject map-based lookups in tests
- `deploy.Config.Validate()` enforces adapter-specific required env vars — Turso needs URL+token, Postgres needs DSN, both need runner URL+service token
- Docker Compose and Dockerfiles live in `deploy/` — control-plane, worker, and daytona-runner as separate services
- Smoke check uses `deploy.RunSmoke()` returning `SmokeReport` with per-service results — CLI exits non-zero on any unhealthy service
- New cloud domain types live in `internal/cloud/` — each table gets its own file (e.g., `run.go`, `attempt.go`)
- SQL schema DDL is exported as package-level `const` strings (e.g., `RunsSchema`, `RunsQueueIndex`)
- Domain structs use `*time.Time` and `*string` for nullable DB columns
- Status enums are `type XStatus string` with `IsValid()` and `IsTerminal()` methods
- Validation follows `func (x *X) Validate() error` pattern checking required fields and enum validity
- Tests use table-driven `modify func(r *Run)` pattern for validation edge cases
- Partial unique indexes (WHERE clause) enforce at-most-one-active invariants
- Foreign key references use `REFERENCES <table>(id)` inline in column definition
- Immutability enforced via BEFORE UPDATE/DELETE triggers with `RAISE(ABORT, ...)`
- Events table has no status enum — event_type is a free-form TEXT column (not an enum)
- Idempotency keys use `TEXT PRIMARY KEY` on `key` column for unique constraint (no separate UNIQUE index needed)
- Domain errors are package-level `var Err*` sentinels with companion `Is*()` helper functions
- Use inline `CHECK (col >= 1)` in DDL for numeric lower-bound constraints (e.g., max_concurrent_runs)
- Lock tables use `released_at IS NULL` partial unique index for active-lock uniqueness (no status enum needed)
- FailureCode is a pure domain enum (no DB schema) — ClassifyFailure() maps codes to retryable/terminal; unknown codes fail safe to non-retryable
- Snapshot-style tables use SnapshotKind enum with IsValid() only (no IsTerminal()) — immutable records don't have lifecycle transitions
- BLOB columns for binary data; UNIQUE INDEX on composite keys as separate DDL constants (not inline table constraints)
- HTTP auth middleware lives in `internal/cloud/authmw/` — Config struct holds JWTSecret/Issuer/Audience and PATAllowlist; Middleware() returns `func(http.Handler) http.Handler`
- Auth middleware uses standard library only (crypto/hmac, encoding/base64) — no external JWT dependencies
- Context keys use unexported `type contextKey int` to prevent collisions; ClaimsFromContext() and IdentityFromContext() for access
- Store interface in `internal/cloud/store.go` is the single persistence contract — service layer must not import adapter packages
- Service-layer components (e.g., SubmitService, ClaimService) accept Store interface + Config struct in constructor; use IDFunc for testable ID generation
- ClaimService uses compensating rollback (TransitionRun claimed→queued, TransitionAttempt→failed) instead of DB transactions for all-or-nothing semantics
- Both adapters' validRunTransitions include `claimed→queued` for rollback support
- Domain errors are sentinel `var Err*` in `errors.go` with `Is*()` helpers; ErrDuplicateKey lives in `idempotency_key.go` (pre-existing)
- Store methods accept `context.Context` first, domain structs by pointer for writes, return by pointer for reads
- Contract tests live in `internal/cloud/storetest/` — adapters call `storetest.Suite(t, factory)` to validate correctness
- Contract test `NewStoreFunc` factory takes `*testing.T` for test-scoped resource management (cleanup, temp dirs)
- Postgres adapter lives in `internal/cloud/postgres/` — one `store.go` (full Store impl), one `schema.go` (Postgres-specific DDL)
- Postgres DDL uses TIMESTAMPTZ (not TIMESTAMP), BOOLEAN (not INTEGER), BYTEA (not BLOB), and NOW() (not CURRENT_TIMESTAMP)
- Postgres immutability uses `CREATE OR REPLACE RULE ... DO INSTEAD NOTHING` instead of SQLite's BEFORE triggers with RAISE
- ClaimRun uses `FOR UPDATE SKIP LOCKED` in a subquery inside UPDATE ... RETURNING for atomic one-winner semantics
- Postgres adapter `isUniqueViolation` matches error code "23505" or "duplicate key" via strings.Contains (no driver import)
- Integration tests use per-test Postgres schemas (`CREATE SCHEMA test_<nanos>_<pid>`) for isolation, with CASCADE drop on cleanup
- Adapter tests are tagged `//go:build integration` and require `HAL_TEST_POSTGRES_DSN` env var
- Turso adapter lives in `internal/cloud/turso/` — reuses domain DDL from `internal/cloud/` (no adapter-specific DDL)
- Turso `ClaimRun` uses `UPDATE...WHERE id=(SELECT...LIMIT 1) RETURNING` — WAL single-writer lock ensures atomicity
- Turso tests use `modernc.org/sqlite` (pure Go, no CGO) with file-based databases and DSN-level PRAGMAs
- Turso `isUniqueViolation` matches `UNIQUE constraint failed` (vs Postgres `23505`)
- Turso `TransitionRun` checks DB status before state machine to give ErrConflict priority over ErrInvalidTransition
- Service-layer lease-loss handling: on ErrLeaseExpired, emit event + mark attempt terminal, then return ErrLeaseExpired (ignore side-effect errors)
- Reconciler/batch-processing services return result structs with counts + per-item error slices for partial-failure reporting
- Attempt struct doesn't carry auth_profile_id — reconciler must GetRun to resolve it for lock release
- RetryService.EvaluateRetry accepts FailureCode as explicit param from caller — avoids needing ListAttemptsByRun store method
- Shared test types (transitionRunCall, transitionAttemptCall) are defined once in claim_test.go — other test files in the same package reuse them
- TimeoutService follows ReconcilerService pattern: ListOverdueRuns → per-run transition + event emission with partial-failure reporting
- validRunTransitions now include `queued→failed` and `retrying→failed` for timeout enforcement — any future status-affecting reconciler should check/update the transition map
- ListOverdueRuns filters by `deadline_at IS NOT NULL AND deadline_at < ? AND status NOT IN (terminal statuses)` — mirrors ListStaleAttempts filtering pattern
- Adding a column to Run requires updating 7 places: run.go (struct+DDL), postgres/schema.go, both adapter scanRun, all SELECT/INSERT/RETURNING queries, and Store method count
- CancellationService: RequestCancel sets flag, CheckAndCancel reads flag + terminates attempt/run + releases lock — worker calls CheckAndCancel every heartbeat
- HeartbeatService checks auth profile revocation status between attempt heartbeat and lock renew — revoked profile emits profile_revoked event, marks attempt failed, releases lock, returns ErrProfileRevoked
- Claim exclusion enforced at SQL level (`WHERE cancel_requested = FALSE/0`) — no service-layer filtering needed
- Shared test types (releaseAuthLockCall) defined once in reconciler_test.go — other test files in same package reuse them
- Runner interface lives in `internal/cloud/runner/` — worker code depends only on the interface, never imports Daytona SDK
- Runner HTTP client uses httptest.NewServer for tests — no external test dependencies needed
- Daytona runner service (TypeScript) lives in `daytona-runner/` — uses Hono framework, @daytonaio/sdk, vitest for tests
- Runner service API must match Go client contract: POST /sandboxes (201), DELETE /sandboxes/{id} (204), GET /health (200), error responses use `{error, error_code}` JSON format
- Service token auth middleware validates X-Service-Token header — returns 401 with error_code "unauthorized" on failure
- Daytona SDK: `daytona.create({image, envVars})` → Sandbox, `sandbox.delete()` for destroy, `sandbox.info()` for status
- Daytona SDK exec: `sandbox.process.executeCommand(cmd, cwd?, timeout?)` → `{exitCode, result}` — no separate stderr in SDK
- Daytona SDK logs: `sandbox.process.getSessionCommandLogs(sessionId, cmdId, onLogs?)` — callback overload for streaming, string overload for batch
- ProvisionService and TeardownService depend on both Store (persistence) and Runner (sandbox API) interfaces — follows same constructor pattern as other services but with dual dependencies
- TeardownService.Teardown takes sandboxID as explicit param (not looked up) — caller has it from provision result or attempt record
- Adding a Store method requires updating 5 places: store.go (interface), store_test.go (mockStore + method count), postgres/store.go, turso/store.go, storetest/contract.go
- Event payloads use `sandboxEventPayload` struct with json.Marshal — emitEvent is best-effort (errors ignored) consistent with other services
- ExecutionService: non-zero exit codes are data (not errors) — ExitCode is captured in ExecutionResult for downstream failure classification via ClassifyFailure()
- ExecutionMode enum (until_complete, bounded_batch) uses underscores internally but buildHalCommand maps to CLI hyphen args (until-complete, bounded-batch)
- AuthMaterializationService: Materialize is no-op when secret_ref is nil/empty; Cleanup follows TeardownService always-emit-done pattern
- Shell single-quote escaping: use `'\''` pattern (end quote, literal escaped quote, start quote) for safe secret injection
- PreflightService validates runtime_metadata_json compatibility (OS, arch, CLI version) before provider-specific preflight command — compatibility check runs before exec
- New FailureCode values must be added in 3 places: validFailureCodes map, retryableCodes map, and the const block (meta-tests catch map drift)
- WritebackService detects changed auth artifacts by reading from sandbox and comparing with original secret_ref — uses UpdateAuthProfile optimistic version check
- Writeback follows dual Store+Runner dependency pattern; reads credentials via Runner.Exec (cat), updates via Store.UpdateAuthProfile with version check
- Redaction rules use capture groups for prefix preservation and non-capturing groups `(?:...)` for in-pattern alternation — mixing these up breaks the applyRule prefix logic
- Event redaction uses `redactPayload()` from `redact_pipeline.go` — call after JSON marshal, before InsertEvent; sets `event.Redacted = true` only when secrets are detected
- `RedactingLogReader` wraps `io.ReadCloser` for streaming log redaction — drop-in replacement for `runner.StreamLogs()` consumers
- Idempotent side effects use `<side_effect_type>:<run_id>` key format with GetIdempotencyKey/PutIdempotencyKey — check-before-do-then-store pattern; PRCreator/PRUpdater/PRCommenter function types decouple from external API
- All three PR side effect services (create, update, comment) live in separate files but share the prSideEffectEventPayload struct; void side effects (update) still store result_ref for auditability
- BundleAllowlist and BundleDenylist are package-level `var` slices (not const) — patterns ending with `/**` match recursively via `matchBundlePattern`
- ComputeBundleHash sorts records by normalized path before hashing — each record contributes `path\x00sha256\n` to the SHA-256 digest
- NormalizeBundlePath converts backslashes to forward slashes and applies path.Clean for cross-platform determinism
- IsBundlePathAllowed checks denylist first (deny takes priority), then allowlist — paths matching neither are denied
- SubmitWithBundle validates bundle first, then delegates to Submit, then stores input snapshot and updates run refs — bundle validation errors happen before run creation
- UpdateRunSnapshotRefs is the Store method for updating input_snapshot_id, latest_snapshot_id, and latest_snapshot_version on a run — adding a Store method requires updating 5 places: store.go, store_test.go, postgres/store.go, turso/store.go, storetest/contract.go
- ErrBundleHashMismatch is a domain error sentinel for hash verification failures — follows the same pattern as other domain errors
- Cloud CLI commands live in `cmd/cloud.go` — `cloudCmd` parent with subcommands (submit, status, logs, etc.) registered via init()
- CLI commands use factory injection (`storeFactory`, `configFactory`) for testable dependency injection — tests override factories to inject mock stores
- `classifySubmitError` maps SubmitService error messages to machine-readable error codes via string matching on known patterns
- JSON output uses `cloudSubmitResponse` / `cloudSubmitErrorResponse` structs — `writeJSON` shared helper for indented JSON output
- CLI mock stores in `cmd/cloud_test.go` implement the full `cloud.Store` interface with no-op stubs (unlike `internal/cloud` which uses `mockStore` embedding)
- Adding a Store method requires updating 6 places: store.go, store_test.go (mock+count), postgres/store.go, turso/store.go, storetest/contract.go, cmd/cloud_test.go (cloudMockStore)
- Cloud CLI error commands use `writeCloudError` for consistent error output formatting (JSON or human); `cloudErrorResponse` is the reusable error struct
- Cloud CLI `--follow` commands use `cloudLogsFollowPollInterval` (package-level var) for testable poll intervals — tests override it to 10ms for fast feedback
- `formatEvent` is the shared event line formatter: `<timestamp>  <event_type>  [payload]` — reuse for future event-display commands
- Cancel command uses `CancellationService.RequestCancel` (sets flag) then `store.GetRun` (reads state) — two-step pattern for commands that mutate then report
- Cloud CLI mock `SetCancelIntent` should update `runsByID[runID].CancelRequested = true` for realistic cancel tests (not a no-op stub)
- `runCloudRun` follows the testable extraction pattern: accepts `baseDir`, `storeFactory`, `configFactory`, and `io.Writer` — `baseDir` enables temp-dir testing without chdir
- `collectBundleFiles` walks `.hal/` directory, applies `cloud.IsBundlePathAllowed` for allowlist/denylist filtering, returns records + contents map
- `compressBundleFiles` uses gzip with simple framing (path\x00length\x00content) — the format matches what `SubmitWithBundle` expects via `BundlePayload.Content`
- Dry-run mode (`--dry-run`) returns early before store/network calls — tests verify nil storeFactory succeeds in dry-run
- `cloudRunResponse` struct contains `run_id`, `status`, `bundle_hash` — distinct from `cloudSubmitResponse` which includes `engine` and `auth_profile` instead of `bundle_hash`
- `decompressBundleFiles` is the inverse of `compressBundleFiles` — reads gzip, parses path\x00size\x00content framing, returns `[]bundleFileRecord`
- Pull command local-change detection compares file bytes (not hash) for simplicity — checks all allowlisted files before writing any, refuses as a batch with `local_changes` error code
- Domain error sentinels use underscore format (e.g., `ErrNotFound` prints `"not_found"`) — error classifiers should match both `not_found` and `not found` patterns
- Auth validate follows service-layer mutation pattern: get profile → check conditions → update fields → emit audit event, with failure setting `last_error_code` before returning error
- Mock store `snapshots` map is keyed by runID (not snapshotID) to match `GetLatestSnapshot` access pattern
- Adding a Store method requires updating 7 places: store.go (interface), store_test.go (mock+count), postgres/store.go, turso/store.go, storetest/contract.go, cmd/cloud_test.go (cloudMockStore), and any service test mocks
- Cloud auth CLI commands live in `cmd/cloud_auth.go` — `cloudAuthCmd` parent with subcommands registered via init(); follows same factory injection pattern as other cloud commands
- AuthLinkService follows dual-purpose pattern: creates profile (store write) + emits audit event (best-effort side effect) — audit event failure does not fail the link operation
- Auth import and auth link share the same service pattern: validate → encrypt secret → create profile → emit audit event — `classifyAuth*Error` functions are identical and could be consolidated
- Adding a Store method requires updating 7 places: store.go (interface), store_test.go (mock+count), postgres/store.go, turso/store.go, storetest/contract.go, cmd/cloud_test.go (cloudMockStore), and any service test mocks
- `cloudMockStore` in `cmd/cloud_test.go` has a `locks` map (`authProfileID → *AuthProfileLock`) for auth lock tests — mirrors the `profiles`, `runsByID`, etc. pattern
- Benchmark harness lives in `internal/cloud/storetest/benchmark.go` — uses `func() cloud.Store` factory and `setupAuthProfile func(id string)` callback for adapter-specific setup
- BenchmarkResult JSON contract: exactly 8 keys (scenario, adapter, runs, errors, claim_p95_ms, heartbeat_p95_ms, duplicate_claims, lock_overcommit_violations)
