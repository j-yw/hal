# Review Report: hal/cloud-worker-orchestration-pipeline

Date: 2026-02-13 10:40

## Summary

This session implemented the cloud worker orchestration pipeline end-to-end on `hal/cloud-worker-orchestration-pipeline`, from atomic run claiming through deterministic attempt/run terminalization, snapshot persistence, optional GitHub PR creation, and `hal cloud worker` command wiring. The approach was incremental by user story, with strict state-transition rules, shared helpers for safety (shell quoting, lock release, hashing), and heavy mock-driven testing for cancellation, heartbeat, lease loss, profile revocation, and cleanup paths. The result aligns with PRD goals for deterministic semantics, workflow-aware artifacts, and stable cloud command behavior.

## Issues Encountered

- Attempt numbers were off-by-one after atomic DB increments; resolved by deriving `Attempt.AttemptNumber` from claimed `Run.AttemptCount` and updating claim mocks to represent post-increment runs.
- Sandbox file decoding broke with wrapped/base64 output differences across environments; resolved by stripping all base64 whitespace before decode and matching mocks against shared `base64Cmd`.
- Lease-loss/profile-revoked handling risked duplicate attempt terminal transitions; resolved by letting heartbeat service own attempt terminalization and keeping worker handlers focused on run transition + cleanup.
- Cleanup sometimes failed when auth locks were already released (`ErrNotFound`); resolved with best-effort lock release that treats not-found as non-fatal while still surfacing real errors.

## Tech Debt Introduced

- RunLoop v1 is still single-threaded, so a long `ProcessOne` blocks poll/reconcile/timeout ticks and limits responsiveness under load.
- Worker command wiring relies on mutable package-level factories for test injection, which increases global-state coupling and restoration discipline requirements.
- Some execution-path tests still depend on command-prefix matching (`execOverrides`), which is brittle when command text evolves.

## Recommendations for Next Priorities

1. Add integration scenarios that execute full worker lifecycle flows (run/auto/review) across Postgres and Turso, including cancel, lease-loss, and profile-revoked checkpoints.
2. Evolve RunLoop scheduling to bounded concurrency or non-blocking orchestration so long-running attempts do not starve poll/reconcile/timeout behavior.
3. Promote heartbeat, cleanup, and terminalization outcomes to stronger structured telemetry (metrics + event fields) for production debugging.
4. Continue consolidating cloud command/sandbox helpers to shared primitives (`ShellQuote`, artifact matchers, best-effort lock release) and remove any remaining ad-hoc variants.

## Patterns Discovered

These patterns have been added to AGENTS.md:

- Keep claim semantics adapter-first: Postgres/Turso `ClaimRun` must atomically set `status='claimed'` and `attempt_count=attempt_count+1`, and ClaimService must use returned `Run.AttemptCount` directly (no additional increment).
- Construct worker runtime via `WorkerPipelineConfig` -> `NewWorkerPipeline`; all core dependencies (`Store`, `Runner`, `WorkerID`, `Claim`, `Checkpoint`, `Heartbeat`, `Cancel`, `Execution`, `Snapshot`) are required and validated up front.
- Cleanup/failure handlers should always use `context.WithTimeout(context.Background(), cleanupTimeout)` (not parent ctx) and call `releaseAuthLockBestEffort` so shutdown cancellation and already-released locks do not break terminalization.
- Snapshot identity must use record-based hashing (`ComputeSandboxBundleHash(records)` -> `ComputeBundleHash`) while bundle payloads keep `path\x00size\x00content` + gzip framing for pull compatibility.
- For `cmd` worker tests, override package-level factories via `injectWorkerTestFactories(t)` and restore in cleanup; no-work polling is modeled by mock `ClaimRun` returning `ErrNotFound`.
