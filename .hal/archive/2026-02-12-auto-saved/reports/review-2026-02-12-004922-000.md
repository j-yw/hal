# Review Report: hal/unified-cloud-ux

Date: 2026-02-12 00:49

## Summary

The session implemented a unified cloud UX where core workflows run through `hal run|auto|review --cloud` and follow-up operations are handled by a reduced support surface (`setup`, `doctor`, `list`, `status`, `logs`, `cancel`, `pull`, `auth`). The work emphasized shared resolution and flag behavior, workflow-kind-aware execution/persistence, deterministic artifact handling, and strict output contracts across human and JSON modes. Security hardening was added through non-secret cloud config validation and centralized redaction in output paths, while legacy cloud command surfaces were removed and locked by tests.

## Issues Encountered

- Secret leakage risk in cloud config values (especially URL query parameters) was found and resolved by recursively scanning YAML keys and all string values for secret-bearing tokens/params.
- Introducing required `workflow_kind` caused cross-layer breakage until schema, scan helpers, submit paths, and test fixtures in both Postgres and Turso stores were updated together.
- Cloud wait/detach behavior drifted across commands and was normalized by shared flag validation plus deterministic precedence rules (`--detach` disables waiting, `--wait` forces waiting, otherwise resolved default).
- Runtime initialization defects (factory wiring, dotenv load order, store lifecycle bounds) were fixed by default factory composition, loading `.env` before config resolution, timeout-bound initialization, and explicit DB close/reset helpers for tests.

## Tech Debt Introduced

- `.gitignore` currently comments out `.hal/*` and branch state includes `.hal/archive` artifacts, increasing repository noise risk and accidental runtime-state commits.
- Cloud configuration concerns remain split between user-facing `internal/cloud/config` and deploy/runtime `internal/cloud/deploy`, which still requires clearer boundary documentation and API guardrails.
- Polling/wait behavior relies on package-level interval overrides for tests; a shared backoff/timeout policy is not yet centralized.

## Recommendations for Next Priorities

1. Add end-to-end integration coverage for the full cloud lifecycle (`setup -> run/auto/review --cloud -> status/logs/cancel/pull`) against an ephemeral store to validate real command interplay.
2. Reinstate strict runtime artifact hygiene (`.hal/archive` and other generated state) and add CI checks to prevent accidental commits of local environment/runtime files.
3. Publish migration-focused command docs/help that explicitly map removed legacy cloud commands to current equivalents and include canonical JSON response examples.
4. Create a unified output-contract test matrix (golden human + JSON snapshots) across cloud and auth commands to prevent field/status/redaction regressions.

## Patterns Discovered

These patterns have been added to AGENTS.md:

- Use `RegisterCloudFlags(cmd)` and `ValidateCloudFlags(flags)` on every cloud-capable workflow command to keep flag semantics and `--detach/--wait` conflict handling consistent.
- Always resolve cloud runtime settings through `config.Resolve(ResolveInput{...})` with the fixed precedence chain: CLI > process env > .env (non-overriding) > `.hal/cloud.yaml` > inferred defaults > hard defaults.
- Keep Cobra handlers thin and delegate to testable helpers (`run<Command>` functions) that accept `io.Writer` plus injectable store/config factories to avoid hard-coupled network/database behavior in tests.
- Model cloud enums as typed strings with a validation map and `IsValid()` (e.g., workflow kind, artifact group), and centralize defaults in constructors like `NewArtifactMetadata(kind)`.
- Route all cloud/auth output through redaction-aware chokepoints (`cloud.Redact()` wrappers) and keep JSON contracts stable with camelCase field names.
