# Review Report: hal/cloud-db-runtime-wiring

## Summary

This session completed the cloud DB runtime wiring so cloud and auth commands can resolve real stores by default without manual factory injection. The implementation centralized database/sql driver registration, added store-only config validation, hardened Turso DSN construction with net/url, and introduced a sync.Once-based runtime composition root with test-friendly factory creation. It also added dotenv loading in the cloud deploy env flow and updated env examples/dependencies, with targeted tests and verification passing.

## Issues Encountered

- Problem: sql.Open("pgx", ...) and sql.Open("libsql", ...) depended on driver registration that was not guaranteed. Resolution: added internal/cloud/deploy/drivers.go with blank imports for pgx stdlib and libsql, then ran go mod tidy for new dependencies.
- Problem: Turso DSN string concatenation risked breaking existing query params and auth tokens containing special characters. Resolution: extracted buildTursoDSN using net/url and added tests for no-query, existing-query, and special-character cases.
- Problem: DefaultStoreFactory uses package-level sync.Once, which is hard to reset across tests. Resolution: added newStoreFactory with injected config loader so tests can run against isolated once instances.
- Problem: dotenv support was needed without making .env mandatory. Resolution: added godotenv.Load() in runCloudEnv, ignored missing-file errors, and downgraded other dotenv errors to warnings.

## Tech Debt Introduced

- Config.ValidateStore and Config.Validate duplicate adapter-specific switch logic, so adapter additions must be kept in sync manually.
- internal/cloud/deploy/runtime.go caches defaultDB globally but does not yet expose close/reset lifecycle hooks.
- Driver validation is mostly indirect (OpenStore failures on unreachable hosts) rather than explicit successful integration coverage.

## Recommendations for Next Priorities

1. Refactor validation paths to share one adapter-validation helper and add a parity test that fails if a new adapter is handled by only one method.
2. Add runtime lifecycle APIs (e.g., CloseDefaultStore and ResetDefaultStoreForTest) and use them in tests/command teardown to make DB connection management explicit.
3. Expand verification with integration-tag tests for successful libsql/pgx connections (or add explicit sql.Drivers assertions in unit tests) so driver registration is validated independently of network behavior.
4. Start Simple Mode PRD work with US-001/US-002 (.hal/cloud.yaml schema and config precedence), reusing the established dotenv-first loading and injectable getenv patterns.

## Patterns Discovered

These patterns have been added to AGENTS.md:

- Register production database/sql adapters only via blank imports in internal/cloud/deploy/drivers.go; OpenStore depends on these init-time side effects and command packages should not duplicate driver imports.
- Use deploy.DefaultStoreFactory as the command composition root by wiring each cloud/auth store factory variable in cmd init() with an if-nil guard so tests can still override factories.
- For code that uses sync.Once, create isolated factories with newStoreFactory(func() Config { return LoadConfig(customGetenv) }) in tests to avoid package-level once-state leakage.
- In cloud deploy CLI paths, call godotenv.Load() before LoadConfig(getenv); ignore os.ErrNotExist and warn (non-fatal) on other dotenv errors.
- Construct Turso DSNs with parse-modify-stringify (url.Parse -> Query().Set("authToken") -> String()) and validate in tests by reparsing URL/query instead of raw string matching.
