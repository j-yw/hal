# Review Report: hal/daytona-go-sdk-hard-cut

Date: 2026-02-12 13:50

## Summary

This session completed the hard cut from the legacy HTTP/TypeScript Daytona runner to a single Go SDK path while preserving the existing runner interface. The migration was delivered incrementally across user stories: DAYTONA_* config schema updates, SDK client lifecycle/exec/log/health implementations, removal of HTTP runner artifacts, deploy manifest cleanup, and CI verification wiring. The implementation style favored thin adapters, stable wrapped error prefixes, and deterministic tests using canceled contexts instead of live Daytona dependencies.

## Issues Encountered

- SDK behavior did not match legacy assumptions: sandbox deletion is invoked on the sandbox object and log retrieval returns map[string]any; this was resolved by standardizing on client.Get before operations and extracting logs from the logs field with safe fallback.
- Config migration surfaced split ownership where smoke checks still depended on runnerURL-style inputs after deploy config moved to DAYTONA_*; this was addressed in US-008 by treating empty runner URL as healthy SDK-direct mode.
- Adding the Daytona SDK introduced a large transitive dependency graph and substantial go.sum churn; resolution was to pin v0.141.0 and focus tests on adapter behavior and error semantics rather than full dependency mocking.
- Cloud lifecycle integration tests were previously not enforced in PR CI; this was resolved by adding make test-integration-cloud-lifecycle to the pull_request workflow.

## Tech Debt Introduced

- CreateSandbox currently maps CreatedAt using time.Now() rather than SDK/server-originated creation metadata, which can diverge from actual provision timestamps.
- StreamLogs is compatibility-oriented: it hardcodes default session/command IDs and returns buffered text via io.NopCloser(strings.NewReader(...)) instead of true incremental streaming.
- DAYTONA_SERVER_URL remains as a legacy fallback alias, which keeps backward compatibility but extends dual-config complexity and migration surface area.
- The .gitignore change that comments out .hal/* increases risk of accidentally committing runtime/archive artifacts from local sessions.

## Recommendations for Next Priorities

1. Add a tagged integration suite that validates SDK success paths end-to-end (create -> exec -> logs -> destroy) against an ephemeral Daytona environment, complementing canceled-context failure tests.
2. Replace CreatedAt time.Now() mapping with SDK-backed timestamp data (or an explicit unknown sentinel) and codify expected timestamp behavior in runner tests.
3. Refactor StreamLogs to use typed parsing helpers and real session/command identification so log behavior is less coupled to dynamic map payloads and default IDs.
4. Define and execute a deprecation timeline for DAYTONA_SERVER_URL fallback once deployments are fully on DAYTONA_API_URL.
5. Tighten repo hygiene by explicitly ignoring runtime .hal artifact subpaths and adding a pre-commit/CI guard for accidental archive and local .env file commits.

## Patterns Discovered

These patterns have been added to AGENTS.md:

- Keep cloud runtime environment schema changes centralized in internal/cloud/deploy/config.go: define constants in one const block, load via LoadConfig(getenv), and encode primary-versus-legacy fallback precedence directly in LoadConfig.
- Implement SDK runner operations in internal/cloud/runner/sdk_client.go with a get-then-operate flow (client.Get first, then sandbox service methods), because Daytona delete/process APIs are exposed on *daytona.Sandbox rather than directly on the client.
- Preserve command-layer behavior by wrapping every SDK operation failure with an operation-specific prefix and %w (create sandbox, destroy, exec, stream logs, health) to keep error contracts stable.
- In internal/cloud/runner/sdk_client_test.go, use canceled contexts to trigger deterministic SDK failures for wrapper assertions; daytona.NewClientWithConfig is safe for unit tests because it performs config construction, not network I/O.
- For cloud workflow regressions, gate PRs with make test-integration-cloud-lifecycle in .github/workflows/ci.yml so lifecycle command and JSON contract drift is caught early.
